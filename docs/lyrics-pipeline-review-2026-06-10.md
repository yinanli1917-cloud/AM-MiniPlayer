# nanoPod Lyrics Pipeline Review — Final Report

**Scope and method.** Twelve improvement proposals were produced across three lenses (speed-and-feel, generalization, robustness). Every proposal was then adversarially verified: a second pass tried to break each claim by reading the actual cited code lines. Result: **4 proposals are sound** (correct as written), **8 are risky** (the diagnosis is correct and verified, but one detail of the proposed mechanism was wrong — each comes with a concrete fix), **0 are invalid, 0 are pending verification**. For the risky ones, build the fixed version described below, not the original text.

## 1. Ranked proposals

### Group A — Sound: verified correct as written

#### 1. Give the lyrics preloader an owner (cancellable, and matching the foreground)

The app quietly pre-fetches lyrics for the next 4 queue tracks, but it throws away the handle to that background task (LyricsService.swift:1588), so the task can never be cancelled — when you skip songs, it keeps running full searches for tracks you already left and competes with the current song for the shared 12-connection network pool. It also hardcodes translation off, so a preloaded track can show different lyrics than the same track played directly. The fix stores the task handle, cancels it whenever a new foreground fetch starts (the exact cancel-and-replace idiom the artwork preloader already uses at MusicController+Playback.swift:736-739), adds a mid-track cancellation checkpoint, and passes the real translation setting through so preload and foreground produce identical cache items.
**Files:** LyricsService.swift. **Verify:** swift test + full LyricsVerifier as standing gates; real-app — skip 5 tracks fast, grep the debug log for exactly one pipeline start per song; translation-parity check on a preloaded NetEase track. **Gain:** protects the 3-second lyrics+artwork budget exactly when you skip rapidly, and preloaded tracks finally display the same lyrics and translations as normal plays.

#### 2. Stop the metadata cache from faking its evidence (store the real durationDiff)

When the title resolver saves a result to disk it does not save how closely the duration matched; on replay it fills in "0" — a fake perfect match (MetadataResolver.swift:742). The safety guard that protects against the postmortem-006 wrong-song class keys on that number, so every cached row — including a poisoned one — passes the guard automatically for up to 30 days. The fix stores the real measured duration difference in the cache entry, returns it on replay, and bumps the cache schema version (4 to 5) so all old fabricated rows are flushed once and the cache self-heals.
**Files:** Utils/MetadataDiskCache.swift, Services/Lyrics/MetadataResolver.swift. **Verify:** unit test for the round-trip and the guard; LyricsVerifier run + benchmark twice back-to-back (cold then warm) — results must be identical across both runs, which is exactly the property the fake "0" breaks today. **Gain:** a wrong cached answer can no longer sneak past a check that a fresh search would have failed — first play and every replay face the same scrutiny.

#### 3. Network-outcome ledger: tell "no lyrics exist" apart from "the internet is down"

Today every network error is silently swallowed, so an offline user sees "Lyrics unavailable" — a false statement — after 5-10 seconds of spinner; worse, on a flaky network one provider's "no match" plus six dead requests can write a 24-hour "no lyrics exist" verdict to disk, suppressing lyrics for a full day. The fix counts, at the single HTTP choke point, how many real responses arrived versus how many requests died in transport: zero responses plus some deaths means "network unreachable", shown honestly and quickly; the 24-hour negative verdict is only written when no transport failures occurred; and a silent system network monitor re-fetches automatically when connectivity returns (no popups). One verified detail: the ledger must default-allow when not bound, so the LyricsVerifier CLI behaves unchanged.
**Files:** Utils/HTTPClient.swift, Services/LyricsService.swift, Services/Lyrics/LyricsFetcher.swift. **Verify:** unit tests for the error-classification table; full LyricsVerifier (unchanged by construction); Wi-Fi-off gate (honest message within ~3s, cache file diff proves no negative row written, lyrics auto-appear on reconnect); partial-failure gate via /etc/hosts blackholing. **Gain:** honest offline message in 2-3 seconds, automatic recovery, and flaky networks can no longer pin a wrong day-long verdict.

#### 4. Typed source registry: provider rules enforced by the compiler, not by strings

The 8 lyric providers are identified by free-form text compared at 79 places, and accuracy guards are tied to exact spellings — one guard is armed by sniffing the substring "qq" in a log label, so a simple rename would silently disarm it; the scorer also gives unknown sources a silent default, and CLAUDE.md already documents a "SimpMusic" source that does not exist in code (proof the string registry has drifted). The fix is one Swift enum of the 8 sources, each declaring its traits (trust tier, score bonus, admission floors, mirror group, which risk checks apply); all call sites read the declared trait, and the compiler forces every new source through a conscious profile. One review caution: the "romanized lyrics check" trait must be its own per-profile boolean, not derived from the trust tier (AMLL is community-tier but must not get that check).
**Files:** Services/Lyrics/LyricsResultSelection.swift, LyricsFetcher.swift, LyricsScorer.swift, LyricsCandidateSelection.swift, LyricsSourceFetchers.swift, plus the CLAUDE.md table fix. **Verify:** oracle-equality unit tests (new profile values versus old hardcoded ladders for all 8 sources), per-case benchmark JSON diff with zero changes, and real-session log evidence that the QQ quarantine and romanization rejection still fire. **Gain:** guards can never silently detach from the provider they protect, two drifted detector copies become one, and adding a source becomes one compiler-checked profile.

### Group B — Risky: correct diagnosis, build the fixed version

Ranking below weighs user value against the risk that remains after the fix is applied.

#### 5. Display-state machine: show lyrics you already have instead of a spinner

The lyrics page decides what to draw from two loose flags, and "loading" always wins — so even when good cached lyrics were just applied, the code flips back to loading for a word-level refetch and you stare at a spinner for 2-5 seconds (LyricsService.swift:511-526); a miss then ends in a late, unexplained ejection from the page. The fix replaces the flags with one published state enum (searching / deep-searching / content / no-lyrics): cached lyrics render immediately with a small refresh indicator while the better fetch continues, and long searches get a visible "searching more sources" phase. **Required fix from review:** deep-searching may only replace searching, never any content state — otherwise the change would hide already-visible unsynced lyrics during the background search, recreating the very bug it fixes.
**Files:** Services/LyricsService.swift, UI/LyricsView.swift (the pipeline itself is untouched). **Verify:** pure-function unit tests for transitions; LyricsVerifier output must be byte-identical to baseline; real-app scenarios including the added fourth one — a Genius/lyrics.ovh-only track must keep its lyrics visible while the deep search runs. **Gain:** cached tracks show lyrics in ~0 ms instead of a 2-5 s spinner, and the no-lyrics wait becomes labeled progress instead of an anonymous spinner.

#### 6. Bounded deadline for the miss path (corrected budget arithmetic) — one work item with #7

A track with no lyrics anywhere holds the spinner ~18 seconds because the background "deep search" has no overall deadline and one child task (the alias-witness search) was never wrapped in a timeout — internally it can chain dozens of serial 2.8 s network calls. The fix wraps that child like its siblings, adds a sentinel (an alarm-clock task that cancels the whole group), and lets provider-confirmed "no lyrics" markers count as an empty result so the existing 2.2-2.95 s fast exit applies. **Required fixes from review:** size the sentinel above the longest legitimate chain (the album-scoped path is 7.7 s, so use ~8-10 s, or wrap each composite child end-to-end); size the witness wrapper as discovery cap plus probe time (~3 s + 6 s) and parallelize discovery across its independent passes; guard availability persistence on cancellation so a truncated search can never write a 24-hour verdict; and use the unclamped evidence windows when treating marker-only results as empty.
**Files:** Services/Lyrics/LyricsFetcher.swift, Services/LyricsService.swift (phase publishing folds into #5's state machine). **Verify:** stub-source unit test asserting the backfill returns within budget; full benchmark plus zh/ja/ko regions with zero pass-rate regression; a slow-network gate (injected latency) proving the album-scoped child still lands — the healthy-network benchmark cannot show this clipping; real-app timing on a known no-lyrics track. **Gain:** worst-case "no lyrics" answer drops from ~18 s to a hard single-digit bound, confirmed misses exit in ~2-3 s, rapid skipping kills the old search instantly.

#### 7. Uniform deadline discipline — merge into #6

This proposal found the same two deadline holes as #6 and contributes two extra pieces: parallelizing the alias-discovery search loop while preserving result order, and the insight that evidence-only markers bypass the fast exit. Review found four defects (sentinel too small, discovery still serial across passes, clamped evidence windows, no cancellation guard on persistence) — all four are covered by the fixes listed under #6. Implement #6 and #7 as one change with the union of both improved versions; do not land them separately. Add one unit test asserting the parallel alias-discovery merge produces the identical alias order as the serial code for a fixture response set.

#### 8. Verdict memoization: stop computing the same answer 12+ times per fetch

During the latency-critical first 2 seconds of every fetch, the "which result wins" function runs 12+ times, and each run re-tokenizes lyric text from scratch in an all-pairs comparison. The fix is memoization — compute once, reuse the answer: a write-once token box on each result (valid because lyric text is immutable) and a per-result solo verdict computed once on arrival instead of on every loop event; both are provably output-identical. **Required fix from review:** add the cancellation guard before availability persistence (same guard as #6), because this change makes background-task cancellation routine; also note the proposal's preload part is already fully covered by #1 — skip it here. The delicate part is splitting the exit-decision closures into cached versus event-dependent terms — land that split as its own commit gated on the benchmark identity diff.
**Files:** Services/Lyrics/LyricsResultSelection.swift, LyricsFetcher.swift. **Verify:** equivalence unit test (memoized versus unmemoized selection on recorded fixtures); full LyricsVerifier identity diff; perf gate — selectBestResult log lines per fetch must drop from 12+ to at most 4. **Gain:** removes heavy repeated text processing from exactly the window where artwork, lyrics, and UI animation compete for CPU, and cuts log spam that buries real traces.

#### 9. Cache the Chinese-resolver results too (tier-separated keys) — extension of #2

The resolver path that serves Chinese-catalog songs best is the only one with zero caching: every replay refires 6 network requests, and after an app restart the full 15-call wave repeats — inside your 3-second perception budget. The fix persists Chinese-tier resolutions with the same real-evidence tuple as #2 and lets each tier replay only rows it produced. **Required fix from review:** the disk cache keys rows by title+artist+duration with no tier component, so CN rows and multi-region rows for the same song would overwrite each other forever — each replay refiring the other tier's full network wave, a regression for exactly the dual-wave pinyin songs this targets. The key must gain a tier discriminator (or CN rows live in their own dictionary) before anything else; the dead preflightExact code removal and the debounced cache rewrite stay as proposed.
**Files:** Utils/MetadataDiskCache.swift, Services/Lyrics/MetadataResolver.swift; build after #2 (shared schema bump). **Verify:** cold-then-warm double benchmark with identical results and a measurably faster warm resolver; the dual-wave pinyin case must show both disk hits and zero refires on the second run; real-app restart test against the 3-second budget. **Gain:** second plays and post-restart plays of Chinese-catalog songs skip all resolver network calls — that stage drops from seconds to ~0 ms.

#### 10. One shared definition of "a real lyric line" (complete, grep-derived site list)

Different filters disagree about which lines count as lyrics — the scorer strips 7 decorative forms (like "。。。" and "···"), most gates strip only 3 — so the same song can flip between lyrics, no-lyrics, or different sources across replays depending on which gate judged it. The fix is one character-class rule (a line counts if it contains at least one letter, digit, or CJK character) replacing all hand-enumerated lists. **Required fix from review:** the original inventory missed at least 5 more gate sites in LyricsSourceFetchers.swift and 3 translation gates in LyricsService.swift; migrating only part of them would recreate the very disagreement this fixes — the site list must be grep-derived and migrated completely in one change, with a regression test asserting no old 3-form literal remains outside the UI layer, plus a targeted before/after run on real tracks that carry leading "♪"/"。。。" lines (the standard 115 cases may contain none).
**Files:** Models/LyricModels.swift (new shared predicate) plus LyricsScorer.swift, LyricsResultSelection.swift, LyricsFetcher.swift, LyricsParser.swift, LyricsSourceFetchers.swift, LyricsService.swift. **Verify:** TDD unit tests for the predicate; per-case benchmark diff requiring identical matched song and source; full-text --dump inspection of any changed case; one real in-app spot check. **Gain:** songs with decorative symbol lines stop flickering between verdicts across replays, and new provider decoration styles are absorbed with zero new code.

#### 11. Japanese reading corroboration replaces the 8-word whitelist (exact-equality matching)

The only way a romaji-titled Japanese song ("hatsukoi") matches its kanji title (初恋) today is a hardcoded 8-word dictionary — a whitelist, which your iron rule bans; the root cause is that the transliteration helper produces Mandarin pinyin readings for kanji, never Japanese readings, so Japanese candidates score zero at every corroboration door. The fix adds a real Japanese-reading function (CFStringTokenizer's Latin transcription — offline, no popups, no latency cost) and teaches the one shared corroboration helper to try both readings. **Required fix from review:** the deleted dictionary had exact-equality semantics, and replacing it with a 0.6 fuzzy threshold inside isTitleMatch admits prefix-extended different songs ("namida" versus 涙色 scores 0.667) — so isTitleMatch must require reading equality; the 0.6 fuzziness stays only at the already-battle-tested ranking doors; the containment shortcut gets floored; and negative fixtures for the prefix-extension class are mandatory.
**Files:** Utils/LanguageUtils.swift, Services/Lyrics/LyricsCandidateSelection.swift, MetadataResolver.swift, plus schema bumps to flush old rows. **Verify:** unit fixtures — the 8 dictionary pairs become test cases (the correct home for known examples) plus the negative prefix pairs; benchmark --region ja and zh before/after with zero pass-to-fail; full-text --dump on every changed case; one real in-app play of a romaji-titled track outside the 8 words. **Gain:** Japanese title matching generalizes from 8 hand-picked words to the whole language while deleting a whitelist from production code.

#### 12. TimelineProfile: compute timeline facts once (Stage 1 only)

Three separate judges decide "do these timestamps plausibly cover this song", with formulas duplicated verbatim and already drifted (a 0.15 versus 0.16 ratio) — so lyrics can pass one judge on first play and fail another on replay. The original Stage 2 (one order-independent rule table) was proven internally inconsistent: the current veto logic is an ordered decision list, and a "take the maximum allowance" model provably admits an album-tagged wrong-master case that today's early branch correctly rejects. **Required fix from review: ship Stage 1 alone** — one TimelineProfile struct computed once per result, with all three judges reading facts from it and every branch and its order kept byte-identical; this is pure de-duplication with provable zero behavior change. The bundled catalog-marker sniff replacement is a separate semantics change and must land on its own, if at all.
**Files:** Services/Lyrics/LyricsScorer.swift, LyricsResultSelection.swift, LyricsFetcher.swift. **Verify:** formula-equality property tests (new allowance equals the legacy expression for durations 60-600 s); full LyricsVerifier per-case diff with zero changes. **Gain:** removes the duplicated, drifted formulas and the hot-loop recomputation, and lays an honest single fact source for any future unification.

## 2. Diagnostics system audit

**What it captures today.** Two layers exist. DiagnosticsService keeps bounded in-memory buffers (300 events, 120 incidents, 120 interactions, 2,400 line-motion samples, 4,000 wave samples, 24 h retention), saves them to a rolling JSON snapshot, and exports per-report bundles with a summary, CSVs, and media. DebugLogger is a plain append-only text log, redirected to a "Live" file whenever owner diagnostics is on. Render-glitch capture works at three granularities: per-frame parity recorders (planned versus applied row position/opacity/blur), 10-second aggregate summaries, and 0.20-0.25 s motion-geometry samples that can raise incidents.

**The log-growth problem.** The Live debug log has no rotation, cap, or truncation anywhere — it is currently about 80 MB (594,105 lines) and still growing; `clearLog()` exists but is never called. Each log call opens a fresh file handle without append mode and without I/O locking, so concurrent writers tear each other's lines — the file contains invalid UTF-8 (this is why grep needs `-a`). The "current session only" gate on report bundles checks only the file's modification time, which is always fresh for an append-only file, so every bundle embeds the entire history (the latest holds a 77 MB log copy), and the Reports folder is never pruned — 75 bundles, 386 MB.

**The gaps that matter for lyric-glitch debugging.**
- Parity errors never become incidents — they dissolve into anonymous P95/max numbers inside a 10-second summary, with no record of which line, when, or what was playing.
- Parity only checks "plan was applied to the layer", not "the plan was right" — a wrong y/blur computed by the engine and applied faithfully shows zero error; an in-code comment admits clipping and blank rows slipped through every green diagnostic.
- The sampling cadences are structurally blind to 1-2 frame (16-33 ms) glitches: motion samples arrive every 200-250 ms, and standalone stall incidents require a 125 ms gap (about 7-8 dropped frames) — exactly the glitch duration you report is below every detector.
- Frame-stall ticks measure the music-position timer, not the lyrics render loop, and the enable flag is read once at startup (toggling needs a relaunch).
- Persistence rewrites a full pretty-printed JSON of all buffers on a 250 ms debounce — continuous serialization work during active playback.
- Wave-timeline samples never reach report bundles at all (absent from the manifest and the snapshot).

**Bounded recommendations (each is a small, contained change).**
1. Per-session log files: at session start, rotate the live log to a stamped file and keep only the last ~5 sessions (DiagnosticsService.swift:4124-4143 + DebugLogger.swift).
2. Fix torn writes: one O_APPEND handle for the session, or route writes through a serial queue — the correct idiom already exists in the same codebase for the CSV writers (DiagnosticsService.swift:640-647).
3. Attach only the current session's log to bundles (automatic once logs are per-session), replacing the ineffective mtime gate at DiagnosticsService.swift:4089-4122.
4. Prune Reports at session start: keep the newest ~10 bundles, delete the rest.
5. Turn parity errors into incidents: when row-y or opacity/blur error exceeds a small threshold for 2+ consecutive frames, raise an incident carrying the row, timestamp, and playback position (consumers of NativeLyricsUXMetrics.swift:684-703).
6. Keep a ~2-second ring buffer of applied per-row values and dump it into any lyric incident — you cannot sample your way to a 16 ms event, but cheap recent history snapshotted on trigger gives after-the-fact evidence.
7. Feed frame-stall ticks from the renderer's own CVDisplayLink, and re-read the enable flag on toggle.
8. Cheapen persistence (drop pretty-printing, exclude motion samples from the rolling snapshot — they already live in the CSV) and add wave samples to the report manifest.

One honest limitation to record: verifying the plan itself (was the computed position correct for this timestamp?) cannot be done by echoing the plan back, and true screen-truth capture would be a larger separate project. The ring buffer plus incident thresholds is the bounded substitute.

## 3. Suggested implementation order

**Step 0 — two tiny prerequisites, land first and alone.**
- The shared cancellation guard: backfillAuthoritativeLyrics persists "no lyrics" verdicts with no cancellation check (LyricsFetcher.swift:1497-1526) — three independent reviews (#6, #7, #8) hit this same hole. Add `guard !Task.isCancelled` before the persistence tail, mirroring the guard fetchAllSources already has at LyricsFetcher.swift:1192-1195. A few lines, fixes a live cache-poisoning vector today, and unblocks three proposals.
- Diagnostics log hygiene (recommendations 1-4 above): do these early because every later verification reads this log, and today it is 80 MB and corrupted.

**Phase 1 — the four sound proposals, one commit each:** #1 preload ownership, #2 durationDiff fidelity (these two are the smallest), then #3 network ledger, then #4 source registry (largest mechanical surface, best-gated).

**Phase 2 — user-facing latency:** #5 display-state machine first (safest of the risky set after its one-guard fix, biggest everyday win), then #6+#7 as a single merged deadline change — its "deep searching" phase plugs into #5's state machine, which is why this order.

**Phase 3 — performance:** #8 memoization; token memo first, the exit-closure split as its own benchmark-gated commit.

**Phase 4 — accuracy-touching generalizations, strictly one at a time with full gates:** #9 CN caching (after #2), #10 comparable-line predicate, #11 Japanese reading, #12 TimelineProfile Stage 1. These touch matching semantics, and the project's own history (postmortem 005: a simplification passed the suite and broke 9 songs) says isolate and verify each.

**Standing rule for every phase:** never bundle two proposals in one commit; each lands with swift test, LyricsVerifier run + benchmark per-case diff against baseline, full-text --dump inspection of any changed case, and the real-app playback check the project rules require. The deeper diagnostics work (parity incidents, ring buffer, wave manifest) is independent of the pipeline and can proceed in parallel at any point.
