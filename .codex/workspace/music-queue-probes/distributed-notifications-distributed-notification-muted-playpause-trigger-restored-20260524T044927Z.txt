## Interpretation checklist

- A no-notification run is only an inconclusive passive capture; rerun while manually causing Music.app to emit a playerInfo event.
- A playpause-restore run is evidence for notification payload shape, not queue editing. It should restore the original play/pause state and, when muted, the original Music.app volume. It must never mutate the queue.
- Metadata/context-only keys support using distributed notifications as invalidation signals, not as queue/history row data.
- Row-carrier keys are not exact proof by name alone; their shapes and values must expose every visible Up Next/history row by order and identity.
- Do not use private notification frameworks or broad system notification scraping as an App Store queue source.
