## Interpretation checklist

- Context label is only a test label; it is not proof of playback origin.
- Compare the visible Music.app Up Next panel manually against neighbor[...] rows.
- Compare browser_window[*].view.neighbor[...] and playlist_window[*].view.neighbor[...] rows when testing whether public window views expose the visible queue.
- Compare application.selection[...] and window selection rows when testing whether Music.app exposes visible queue selections through public AppleEvents.
- A window view surface is still only a public playlist view; it is not exact queue proof unless it exposes every visible history/current/upcoming row by order and identity.
- A selection surface is selected visible items only; it is not exact queue proof unless it exposes every visible history/current/upcoming row by order and identity.
- If Play Next / Play Later edits appear in Music.app but not in this snapshot, currentPlaylist scanning is not a real queue mirror.
- If radio/station playback has no current playlist tracks or unrelated neighbors, mark that context unavailable.
- Treat ApplicationMusicPlayer.queue as app-local unless a separate proof shows it targets the same Music.app playback session on macOS.
- For focused ApplicationMusicPlayer runtime evidence, run .codex/workspace/probe_musickit_application_player_queue.sh.
- Treat SystemMusicPlayer as unavailable on macOS unless the local SDK compiler probe changes.
- Treat MediaPlayer MPMusicPlayerController as unavailable on macOS unless the local SDK compiler probe changes.
- Treat MPNowPlayingInfoCenter as current-application now-playing metadata, not system Music.app queue state.
- Do not treat Apple Music recently played API data as live queue proof.
